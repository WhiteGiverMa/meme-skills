# 梗技能创建器

此 skill 用于在 meme-skills 知识库中创建新的热梗 skill，实现梗知识的持久化记忆与复用。

## 触发判断

### 明确触发（用户主动要求）

用户说出以下类型的表述时，立即进入创建流程：
- 「做个 skill」「创建 skill」「收录这个梗」
- 「记一下这个梗」「保存一下」「存个档」
- 「这个梗也加进去吧」

### 主动提议（Agent 判断）

以下条件**全部满足**时，Agent 应主动提议创建 skill：
1. 当前讨论的梗**不在**已有 `meme-*` skill 覆盖范围内
2. 该梗已有足够的网络传播量（可搜到多个独立来源）
3. 用户在对话中对此梗表现出反复提及或深入讨论

提议话术示例：
> 「这个梗资料挺丰富的，要不要我收录到 meme-skills 里？下次再聊到就能直接调用了。」

不要频繁提议（同一会话中最多提议 2 次）。

## 创建流程

```
触发 → 命名 → 搜索 → 验证 → 编写 → 索引 → 提交
```

### 步骤 1：命名

确定 skill 名称 `meme-<拼音>`：
- 拼音全小写，连字符分隔，不带声调
- 如果梗有多个名称，选最通用的一个
- 示例：奶龙 → `meme-nailong`，你已急哭 → `meme-niyijiku`

向用户确认名称后再继续。

### 步骤 2：搜索（跨来源交叉验证）

对梗的以下维度分别搜索，每个维度至少命中 2 个独立来源：

| 维度 | 搜索目标 | 问题示例 |
|---|---|---|
| 起源 | 创作者、最初平台、最早出现时间 | "XX梗 起源 最早" |
| 传播 | 出圈节点、关键传播事件 | "XX梗 为什么火了" |
| 含义 | 字面义、引申义、使用场景 | "XX梗 是什么意思" |
| 演变 | 变体、衍生、语义漂移 | "XX梗 演变 新含义" |

**推荐搜索来源**：
- 百度百科、萌娘百科（基础事实）
- 腾讯新闻、虎嗅、澎湃新闻（传播分析）
- 单词乎、游侠手游（梗百科类网站）
- B站/抖音相关视频标题与简介（当前热度）

### 步骤 3：验证

创建 skill 前，必须确认：
- [ ] 梗的**起源人物/账号**有至少 2 个独立来源交叉确认
- [ ] **时间线**关键节点有来源支撑
- [ ] **衍生概念**有实际案例引用
- [ ] 没有将「网友玩梗编造」的内容当作事实写入

如果某个关键事实只有一个来源，标注为「待验证」或降低确信度表述。

### 步骤 4：编写 SKILL.md

参考 `references/meme-template.md` 模板，按以下结构编写：

1. **概述**（2-3 句话，给出梗的快照）
2. **起源**（创作者、时间、平台、原始内容、出圈过程）
3. **核心特征/含义**（形式、含义、使用场景）
4. **发展经过/传播时间线**（表格形式，关键时间节点）
5. **衍生概念**（变体、二次创作、相关梗）
6. **文化分析**（社会心理、时代情绪）
7. **Agent 使用指南**（如何识别与回应、注意事项）
8. **参考资料**（每条带可访问 URL）

**编写原则**：
- LLM 训练数据已知的基础信息从简，重点覆盖训练截止后的新演变
- Agent 使用指南必须具体——给出「当用户说 X，TA 想表达 Y，你应该回应 Z」的模式
- 语言：正文用中文，引用 URL 保持原始格式

如果梗的信息极少（搜不到 2 个独立来源），不要直接拒绝——先看用户意图：是要贡献到仓库，还是自己留着用？见下方「仓库标准 vs 本地自用」。

### 步骤 5：更新项目索引

创建 `meme-<name>/SKILL.md` 后，更新以下文件：

**`README.md`**：在「已收录梗」表格中新增一行：

```markdown
| [meme-xxx](./meme-xxx/SKILL.md) | 梗中文名 | 分类 | 关键词1、关键词2、关键词3 |
```

**`AGENTS.md`**：在目录结构中新增条目：

```
├── meme-xxx/               # 梗中文名
│   └── SKILL.md
```

### 步骤 6：Git 提交

```bash
git add meme-<name>/ README.md AGENTS.md
git commit -m "feat: add meme-<name>, <梗中文名>"
```

仅在用户明确要求推送时才执行 `git push`。

## 仓库标准 vs 本地自用

meme-skills 是一个**公共知识库**，收录的每个梗都会被其他人使用。因此仓库对 skill 质量有底线要求。但这不意味着所有梗都必须进仓库——**你自己用的 skill，宽松得多**。

```
┌──────────────────────────────────────────────────────┐
│  这个梗适合放进 meme-skills 仓库吗？                    │
│                                                      │
│  YES → 严格流程：搜索验证 → 完整模板 → 更新索引 → PR  │
│  NO  → 问一句：「要不要在本地留个档？自己用没那么多讲究」│
└──────────────────────────────────────────────────────┘
```

### 不适合贡献到仓库的情况

以下情况**不建议**提交到 meme-skills 仓库，以轻松的语气告知用户：

| 情况 | 原因 | 话术示例 |
|---|---|---|
| 信息不足 | 搜不到 2 个独立来源，关键事实无法验证 | 「网上关于这个梗的资料太少了，放仓库里怕写错。不过你自己用的话随便写写就行，要搞一个吗？」 |
| 太新了 | 梗刚出现不到 2 周，传播还在演化中 | 「这梗也太新鲜了，现在写可能下周就变味了😂 等它飞一会儿再入库。当然你要想先在本地记个笔记我帮你搞。」 |
| 已有覆盖 | 被现有 skill 覆盖了 | 「这个其实已经在 XXX 那个 skill 里了，要不我去那边补两笔？如果你是想自己记个不同角度的那也行。」 |
| 非中文梗 | 知识库定位是中文互联网迷因 | 「这是英文圈的梗诶，meme-skills 目前只收中文互联网的。不过你自己本地用的话无所谓，要留一份吗？」 |
| 太小众 | 传播范围极窄，只有小圈子在用 | 「这个梗太小圈子了，放公共仓库里别人看到会一脸懵。但你这小圈子的梗自己留着用挺合适的，搞不搞？」 |

### 本地自用模式

如果用户想要本地 skill，切换为**轻量流程**：

1. 跳过严格的交叉验证（一个来源也行，凭记忆也行）
2. 模板可以简化——起源和传播时间线可以粗略，重点是 Agent 使用指南
3. **不要**更新 README 和 AGENTS.md（本地 skill 不进仓库索引）
4. 提醒用户：这个 skill 放在本地 skills 目录就行，下次聊到就能调用

话术示例：
> 「行，不走仓库标准，给你搞个本地速通版——信息可能不够严谨，但你自己用足够了。回头要是梗长大了，再升级成正式版也不迟。」
